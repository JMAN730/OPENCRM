================================================================================
  1. EXECUTIVE SUMMARY
================================================================================

WHAT THE APP DOES
-----------------
OpenCRM is a B2B sales-team CRM targeting high-volume outbound callers. Core
capabilities: lead management, a simulated click-to-call dialer, task
management, a team/member hierarchy, an automated Google Maps lead scraper,
and dashboard analytics.

CORE BUSINESS MODEL
-------------------
Multi-tenant SaaS: each company (Organization) owns isolated data. Currently
no billing layer.

MAIN ARCHITECTURE DECISIONS
----------------------------
  - Next.js 16 App Router with tRPC for type-safe API
  - PostgreSQL via Prisma 7 + @prisma/adapter-pg
  - NextAuth v4 with JWT strategy
  - Feature-based modularization (src/features/<name>/)
  - Docker Compose for full-stack deployment

OVERALL CODEBASE HEALTH SCORE: 6.5 / 10

The structural skeleton is solid and shows real engineering thought. However,
there is an incomplete SQLite->PostgreSQL migration leaking dead dependencies
into production, several stub/empty features shipped as UI placeholders,
missing route-level middleware, no pagination on the primary data table, and a
TODO.txt that still references unfixed UX issues.

BIGGEST STRENGTHS
-----------------
  - Clean feature-based modularization that scales to more engineers
  - Proper organizationId isolation on every query — no obvious tenant-escape bugs
  - End-to-end type safety via tRPC + Zod + TypeScript
  - JWT fallback context recovery is a thoughtful operational resilience pattern
  - Role-scoped lead access (resolveLeadScope) is well-factored and reusable
  - Test coverage exists for every major router and several components

BIGGEST WEAKNESSES
------------------
  1. No route-level middleware active — proxy.ts is never wired as middleware.ts
  2. Incomplete SQLite->PostgreSQL migration — dead @libsql/client +
     @prisma/adapter-libsql dependencies still installed and referenced in
     next.config.ts
  3. No pagination on leads.getAll — will OOM or time out at scale
  4. importRowsToLeads loads all org leads into memory for deduplication
  5. Dashboard fires 14+ parallel DB queries per load with no caching
  6. Multiple stub features (Outreach, partially Analytics) shipped as permanent
     UI placeholders
  7. tasks.update uses protectedProcedure (not organizationProcedure) —
     org-scope isolation gap
  8. inviteUser sets password directly — no email invitation flow, no way for
     invited users to know their credentials

HIGHEST-RISK TECHNICAL PROBLEMS
--------------------------------
  1. Missing route middleware (unauthenticated page loads possible)
  2. Lead importer memory explosion at scale
  3. No rate limiting on auth endpoints
  4. deleteAccount doesn't invalidate JWT — ghost sessions persist 30 days
  5. Hardcoded SQLite comment in PostgreSQL schema misleads future developers

================================================================================
  3. ARCHITECTURE BREAKDOWN
================================================================================

3.1 Frontend Architecture
--------------------------
Next.js 16 App Router. All authenticated pages are "use client" and wrapped in
<DashboardLayout>. Feature components live in src/features/<name>/components/.
There is no server-side rendering of user data — all fetching is client-side
via React Query/tRPC hooks.

ISSUE: Because all pages are client-rendered, an unauthenticated user who
reaches /leads will see a loading spinner or empty state — not a 401. Route
protection relies entirely on the missing middleware (see Section 8).

3.2 Backend Architecture
-------------------------
Pure tRPC over HTTP POST. No REST endpoints except NextAuth's /api/auth/*.
All business logic is in src/features/<name>/server/router.ts files merged
into appRouter at src/server/api/root.ts.

3.3 Auth Flow
--------------
  Sign-in form -> NextAuth CredentialsProvider.authorize()
    -> DB lookup (email OR name) -> bcrypt.compare()
    -> JWT issued with { id, email, role, organizationId, teamId }
    -> 30-day JWT stored in cookie
    -> tRPC context reads via getServerSession() or JWT fallback

CRITICAL GAP: After deleteAccount, the JWT is not invalidated. User can
continue making tRPC calls for up to 30 days. The protectedProcedure would
succeed (session exists) but DB operations would fail with FK errors (user
deleted). This is an unhandled error path that would return confusing 500s.

3.4 Database Structure
-----------------------
See Section 4 for full analysis. Summary: PostgreSQL 16, Prisma 7, schema push
(no migrations). 13 models. The schema is clean but retains SQLite-era
artifacts.

3.5 API Structure
------------------
Single tRPC endpoint at /api/trpc/[trpc]. Supports GET (query) and POST
(mutation). Batch requests via httpBatchLink. Root router has 7 namespaces:
leads, calls, scraper, tasks, dashboard, auth, teams.

3.6 State Management
---------------------
React Query (via tRPC). No global state store (no Zustand/Redux).
Component-local state via useState. This is appropriate for the app's current
complexity.

3.7 Caching Strategy
---------------------
  - Client: React Query default staleTime (0 — refetches on mount). Sidebar
    counts set staleTime: 30_000.
  - Server: NONE. No Redis cache wired despite ioredis being installed and
    src/lib/redis.ts existing.
  - Dashboard KPIs are recomputed on every page load.

3.8 File Organization
----------------------
  src/
    app/                  # Next.js App Router (pages + API routes)
    features/             # Feature modules (self-contained)
      <name>/
        components/       # React components
        server/router.ts  # tRPC router
    server/               # Shared server utilities
      api/root.ts         # Root router
      trpc.ts             # Context + procedures
      activity.ts         # Activity logging helper
      teams/scope.ts      # Lead scope resolution
      scraper/            # Scraper subsystem
    components/           # Shared layout + shadcn/ui
    lib/                  # Singletons (auth, prisma, email, redis, utils)
    hooks/                # React hooks
    types/                # Type augmentation
    test/                 # Test utilities
    proxy.ts              # Route guard (ORPHANED — see Section 8)

3.9 Deployment Setup
---------------------
Docker Compose with three services: db (PostgreSQL 16), migrate (one-shot
prisma db push), app (Next.js standalone). depends_on with health checks is
correctly configured. Volume scraper_output is shared between host and
container.

3.10 Real-time Systems
-----------------------
NONE. No WebSockets, SSE, or polling. Scraper job status is polled by the UI
via useQuery with a refetchInterval. This is simple and correct for current
scale.

3.11 Background Job Systems
----------------------------
Scraper jobs run as OS child processes via child_process.spawn. No queue. No
persistent worker. State is tracked in-memory (globalForScraper.__scraperRunning).
A PM2 restart or container restart will mark all running jobs as FAILED
(handled by reconcileOrphanedJobs — good), but in-progress scraping work is
lost.

3.12 Permissions/Roles System
------------------------------
Three roles: ADMIN, MANAGER, USER.

  - ADMIN:       Full org access, team CRUD, member management, all leads
  - MANAGER:     Role exists in enum but has NO SPECIAL BEHAVIOR in any router
                 — treated identically to USER for all non-admin checks
  - USER:        Sees only assigned leads (or team leads if a team leader)
  - Team Leader: User with leaderId in a team — gets team-scope lead visibility

GAP: MANAGER role has no distinct privileges, making it a dead value in the
enum. Either define permissions for it or remove it.


================================================================================
  4. DATABASE + PRISMA AUDIT
================================================================================

4.1 Schema Quality: Good with Issues
--------------------------------------

STRENGTHS:
  - Proper organizationId on all tenant-scoped models
  - createdAt/updatedAt on all models
  - Selective indexes on hot query paths
  - onDelete: Cascade for owned rows, SetNull for optional relations
  - cuid() PKs (sortable, URL-safe, no UUID collision overhead)

PROBLEMS:

P1 — ScraperJob.locations/categories as JSON strings (schema.prisma:294)
-------------------------------------------------------------------------
  locations  String @default("[]")
  categories String @default("[]")

The inline comment says "SQLite does not support native array columns" but the
schema is now PostgreSQL. These should be String[]:

  locations  String[] @default([])
  categories String[] @default([])

This removes the serialization/deserialization boilerplate throughout runner.ts
and scraperRouter.

P2 — LeadTag has no FK to Organization (schema.prisma:151)
----------------------------------------------------------
  model LeadTag {
    organizationId String   # raw string, no @relation
  }

No Prisma relation, no @@index, no cascade. The organizationId field is
orphaned — there's no way to enforce org-scoping or cascade deletes. This
model is entirely unused in any router.

P3 — ScraperJob missing onDelete: Cascade on Organization relation
  (schema.prisma:311)
------------------------------------------------------------------
  organization Organization @relation(...)  # no onDelete
  userId       String                        # no onDelete

Deleting an Organization with existing scraper jobs will throw a FK constraint
error. The auth.deleteAccount procedure manually deletes scraper jobs first —
a workaround for a missing cascade.

P4 — CallLog.userId cascades on User delete (schema.prisma:187)
---------------------------------------------------------------
  user User @relation(... onDelete: Cascade)

If a user is deleted, all their call logs are deleted too. This destroys
historical call data. Should be SetNull with userId String?.

P5 — Activity.userId cascades on User delete (schema.prisma:238)
----------------------------------------------------------------
Same as P4 — audit trail is destroyed when a user account is deleted. Should
be SetNull.

P6 — No index on Task.userId alone (schema.prisma:261)
------------------------------------------------------
  @@index([userId, dueDate])

getAll queries where: { user: { organizationId } } — this hits a relational
join, not an index on Task directly. A @@index([userId]) or direct
organizationId field on Task would be faster.

P7 — Duplicate KPI computation in dashboardRouter
-------------------------------------------------
  appointmentsSetCount:  prisma.lead.count({ where: { organizationId, status: "CONNECTED" } })
  qualifiedLeadsCount:   prisma.lead.count({ where: { organizationId, status: "CONNECTED" } })

Exact same query runs twice in the same Promise.all (lines 30 and 35 of
router.ts).

P8 — No index on Lead.assignedToId
-----------------------------------
The scope resolution queries where: { assignedToId: { in: [...] } } but there
is no @@index([organizationId, assignedToId]). At 100K leads, scope queries
will full-scan.

4.2 Unused Models
------------------
  - LeadTag       — defined, never queried
  - Pipeline      — defined, never queried (FK on Lead exists)
  - PipelineStage — defined, never queried

4.3 Naming Consistency
-----------------------
  - CallLog vs calls (router/table name mismatch — cosmetic)
  - Activity.type is a String not an enum — "LEAD_CREATED", "NOTE_ADDED" etc.
    are stringly typed. Should be a Prisma enum.

5.9 tasks.update Org Isolation — GAP

  update: protectedProcedure  // <- not organizationProcedure
    .input(...)

      if (!task || task.userId !== ctx.session.user.id) throw NOT_FOUND;
      // No organizationId check

A user can update/complete any task owned by themselves, regardless of org.
The current check task.userId === ctx.session.user.id prevents cross-user
access but not cross-org access, and violates the design contract that all

================================================================================
  6. TEAM SYSTEM AUDIT
================================================================================

6.1 What Works
---------------
  - ADMIN creates teams, sets leader, adds members (inviteUser + setMembership)
  - Team leaders see all leads assigned to their team members via
    resolveLeadScope
  - activityFeed filters by team membership with proper authorization
  - memberDetail returns leads, calls, open tasks, and counts for any visible
    member

6.2 What's Broken
------------------

INVITE FLOW (Critical UX Bug):
The inviteUser procedure creates a user with a provided password. There's no
email sent. Admin creates account -> must call/message/email the new user
out-of-band with their credentials. This is not acceptable for a production
SaaS.

NO USER REMOVAL:
There's no procedure to remove a user from the organization. setMembership can
remove from a team (set teamId: null) but there's no way to set
user.organizationId = null or delete a user account as an admin.

MANAGER ROLE IS INERT:
Every RBAC check in teams is isAdmin(role) (string === "ADMIN"). MANAGER gets
no special access. This means the role is a dead enum value.

ACTIVITY FEED AUTHORIZATION BUG:
Activities are filtered by userId: { in: memberIds } — only members, not the
leader. If the leader is not in the users relation (only set via leaderId),
the activity feed may miss the leader's own activities.

N+1 IN activityFeed:
Each call without a teamId runs two team queries (find team, then check
leader) before the activity query.

6.3 Database Concern
---------------------
Team.leaderId is SetNull on user delete. If the leader is deleted, the team
has no leader and no indication of this. A soft-delete or
leadership-transfer requirement would be safer.

================================================================================
  8. SECURITY AUDIT
================================================================================

8.1 Missing Route Middleware (CRITICAL)
----------------------------------------
As documented in Section 5.8, proxy.ts is never activated as Next.js
middleware. Protected routes are not redirected at the edge/server level. Only
tRPC procedures enforce auth.

SEVERITY: Medium-High. Data is protected (tRPC enforces auth), but page
structure is visible to unauthenticated users.

8.2 Ghost Sessions After Account Deletion
------------------------------------------
  // authRouter.deleteAccount
  await ctx.prisma.user.delete({ where: { id: userId } });
  return { ok: true };
  // JWT is still valid for 30 days

After calling deleteAccount, subsequent tRPC calls with the old JWT will pass
enforceUserIsAuthed (session exists) but fail at DB level with confusing
errors.

FIX: Implement a JWT denylist (Redis SET), or force re-auth by reducing
maxAge.

8.3 Login Allows Name as Username
-----------------------------------
  // auth.ts:31-34
  where: {
    OR: [
      { email: credentials.email },
      { name: credentials.email },  // name is not unique
    ]
  }

User.name is not unique in the schema. Two users can have the same name. The
lookup is non-deterministic. bcrypt.compare will still fail for the wrong
user, but this is an avoidable ambiguity and potential subtle auth bug.

FIX: Remove the name OR clause; query by email only.

8.4 No Rate Limiting
---------------------
No rate limiting on:
  - auth/signin (NextAuth) — brute-forceable
  - auth.register — account enumeration + spam
  - auth.resetPassword — email bombing
  - Any tRPC mutation

FIX: Implement Upstash Ratelimit or rate-limiter-flexible in middleware and
auth procedures.

8.5 Password Reset Token Stored Raw
-------------------------------------
  const token = crypto.randomBytes(32).toString("hex");
  await ctx.prisma.passwordResetToken.create({ data: { token, userId, expires } });
  const resetUrl = `${process.env.NEXTAUTH_URL}/auth/reset-password?token=${token}`;

The raw token appears in both the DB and the URL. If server logs are shipped
externally, the token is exposed. Industry best practice: store hash(token),
compare hash(input).

8.6 calls.logCall Missing Scope Check
---------------------------------------
callsRouter.logCall confirms the lead is in the caller's org but does NOT
check if the lead is assigned to the calling user. A regular user can log a
call against any lead in their org — bypassing resolveLeadScope which all
other lead operations use.

8.7 Input Validation: Good
---------------------------
All tRPC inputs are Zod-validated. String length limits enforced. Enum
validation on status fields. No SQL injection vectors through Prisma
(parameterized queries).

8.8 Multi-tenant Isolation: Good
----------------------------------
Every organizationProcedure query filters by ctx.organizationId. The
resolveLeadScope pattern correctly narrows by role. No observed cross-tenant
data leak vectors.


================================================================================
  9. PERFORMANCE + SCALE AUDIT
================================================================================

9.1 N+1 and Inefficient Queries
---------------------------------

leadsRouter.getAll — No Pagination
  return ctx.prisma.lead.findMany({ where: finalWhere, orderBy: ... });
  // No take/skip

Returns every lead in scope. At 10K leads, sends 10K rows to client on every
page load. At 50K, will OOM the browser tab.

importRowsToLeads — Full Table Scan for Dedup
  const existing = await prisma.lead.findMany({
    where: { organizationId },
    select: { company: true, phone: true },
  });

Loads all org leads into Node.js memory. At 100K leads, ~50MB per import
operation. Should use batch-check with indexed compound key or a composite
unique constraint.

dashboardRouter.getKpiStats — 14 Parallel Queries
14 queries fired on every dashboard load with no caching. The 7-day
callsPerDay loop runs 7 identical count queries that could be a single
groupBy(createdAt).

resolveLeadScope — Uncached
Called once per leads query. Fetches team membership from DB every time.
Should be cached in the JWT token or Redis.

activityFeed — Multiple Queries Before Main Query
Each call without a teamId runs two team queries (find team, then check
leader) before the activity query.

9.2 Bundle Size Issues
-----------------------
  Package              Size          Status
  -----------------------------------------------------------------------
  xlsx (SheetJS)       ~600KB min.   Should be dynamically imported
  framer-motion        ~150KB        Used sparingly
  @aws-sdk/client-s3   ~200KB        Unused, installed
  openai               ~200KB        Unused, installed
  @libsql/client       ~2MB          Dead, from SQLite era

9.3 Render Performance
-----------------------
  - LeadsList.tsx renders a full DOM table of all leads — no virtualization.
    At 1K+ rows, scrolling will be janky.
  - Dashboard fires all tRPC queries on mount with no suspense boundaries —
    cascading loading waterfalls.

9.4 Scaling Bottlenecks by User Count
---------------------------------------

100 USERS:
Functional. Lead tables under 10K per org. Dashboard queries fast. Scraper
jobs manageable.

1,000 USERS / ~50 ORGS:
Dashboard stats start to slow (no caching). Lead imports hit memory pressure
if any org has 50K+ leads. resolveLeadScope DB round-trips become noticeable
latency.

10,000 USERS / ~500 ORGS:
leadsRouter.getAll causes client browser OOM for large accounts.
importRowsToLeads dedup scan becomes a minutes-long operation. Dashboard
14-query pattern saturates DB connection pool. Scraper job log writes (every
1.5s per job) create write contention at dozens of concurrent jobs.

100,000 USERS / ~5,000 ORGS:
PostgreSQL needs read replicas. The scraper in-memory registry doesn't work
across multiple Node processes — horizontal scaling is impossible without
Redis/queue. ScraperJob.logs TEXT field with 200K char limit creates large row
updates. Need dedicated logging service (S3, Loki).

  11. FINAL VERDICT
================================================================================

IS THIS APP STRUCTURALLY SOLID?
---------------------------------
Yes, with serious caveats. The core request pipeline (Client -> tRPC -> Router
-> Prisma -> PostgreSQL) is well-structured and type-safe. Multi-tenancy is
properly enforced. The feature-based module system is clean and would support a
team of 5-8 engineers without major collisions.

The caveats are real: a missing middleware file leaves routes unprotected at
the Next.js layer; an incomplete database migration creates a false mental
model; and several advertised features are UI-only placeholders.

WHAT WOULD WORRY A SENIOR ENGINEER MOST?
------------------------------------------
  1. The proxy.ts that was never wired as middleware. This is the kind of bug
     that gets missed in code review, ships to production, and gets exploited.

  2. importRowsToLeads loading all org leads for deduplication. One large org
     doing a bulk scraper import would OOM the Node.js process, taking down the
     entire app for all tenants.

  3. No rate limiting. Auth endpoints are trivially brutable.

  4. Stub features with real UI shipped. Outreach has zero backend. A customer
     clicking "New Sequence" gets a dead button. This destroys trust.

WHAT SHOULD BE FIXED BEFORE SCALING?
--------------------------------------
In strict order:
  1. Wire middleware.ts (30 minutes)
  2. Paginate leadsRouter.getAll (2 hours)
  3. Fix importRowsToLeads dedup (4 hours)
  4. Remove dead libsql packages (30 minutes)
  5. Add rate limiting to auth (1 day)

IS THE ARCHITECTURE CAPABLE OF BECOMING A SERIOUS SAAS?
---------------------------------------------------------
Yes — the foundation is right. tRPC + Prisma + Next.js is a proven stack. The
feature-based organization scales to a larger team. The multi-tenant design is
correct. The main architectural work needed for scale is: a job queue for the
scraper, Redis caching for dashboard queries, read replicas, and proper
pagination.

TOP 5 CRITICAL ACTIONS TO TAKE IMMEDIATELY
--------------------------------------------
  1. Create src/middleware.ts — fix the orphaned proxy.ts

  3. Fix importRowsToLeads dedup — use prisma.lead.count with indexed lookup,

  4. Remove dead SQLite packages — @libsql/client, @prisma/adapter-libsql,
