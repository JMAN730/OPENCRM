1. EXECUTIVE SUMMARY
================================================================================

What the App Does
-----------------
OpenCRM is a multi-tenant SaaS CRM targeting high-volume outbound sales teams —
particularly home-service businesses (landscaping, power washing, mobile mechanics,
etc.). Core value proposition: scrape Google Maps leads → work them in a
dialer-style pipeline → track activity across a sales team.

Core Business Model
-------------------
B2B SaaS CRM with organization-level tenancy. One registration creates one
organization; admins invite users (by setting their passwords directly — no email
invite flow exists yet). Billing, subscription management, and any payment layer
are entirely absent.

Main Architecture Decisions
----------------------------
- Next.js 16 App Router + tRPC v11: solid, type-safe, appropriate for the scale
- JWT sessions (NextAuth v4): stateless, easy to scale horizontally
- PostgreSQL + Prisma v7: canonical, no ORM lock-in risk
- Redis as optional sidecar: smart design — app degrades gracefully without it
- Python subprocess for scraping: pragmatic but couples the app to a filesystem
  and process model incompatible with serverless

Overall Codebase Health Score: 7.2 / 10

  Category                Score   Notes
  ─────────────────────────────────────────────────────────────────────────────
  Auth security           8.5/10  Solid JWT revalidation, constant-time bcrypt, rate limiting
  API design              8/10    Consistent tRPC patterns, good Zod validation
  Database schema         6.5/10  Good indexes, but dual-state fields, nullable FKs, zombie models
  Frontend consistency    6/10    Dual component systems, dead code, simulated features
  Feature completeness    5/10    Multiple stubs, simulated dialer, no billing
  Test coverage           7/10    Good router coverage, gaps in frontend and integrations
  Production readiness    5.5/10  Docker deploy works; billing/invite/dialer are broken
  Performance design      7/10    Redis caching done right; N+1 and client-side compute issues
  Security                7.5/10  Real effort put in; a few gaps that matter at scale

Biggest Strengths
-----------------
1. The auth subsystem is genuinely well-designed: dummy hash timing attack
   prevention, Redis auth-snapshot cache with 60s TTL, token hash (never raw
   token in DB), deleted-user JWT invalidation.
2. tRPC + Zod gives end-to-end type safety from DB to client component.
3. Organization isolation is consistently applied across all 9 routers.
4. Redis is optional and fails open everywhere — app runs without it.
5. Lead scope (resolveLeadScope) is centralized, cached, and correctly applied.
6. prisma db push approach (no migration history) is appropriate for this stage.

Biggest Weaknesses
------------------
1. No billing/subscription system — not a line of code, despite "Billing" tab in
   Settings UI.
2. ✅ Dialer data corruption FIXED — dialer route now shows a disabled page; it
   no longer silently logs every call as CONNECTED.
3. Website generator has no AI — it is three hardcoded templates. OPENAI_API_KEY
   is not wired up. (CHANGELOG no longer claims otherwise.)
4. Real customer data in git history — Leads_SBL_Tampa.xlsx was removed from the
   working tree but is still in full git history. Requires filter-repo scrub.
5. ✅ CHANGELOG false claims FIXED — Twilio/AI analytics claims removed; README
   and CHANGELOG now accurately describe what is and isn't built.
6. ✅ LeadDetailsModal.tsx dead code FIXED — file and its test already deleted.

Highest-Risk Technical Problems
--------------------------------
1. Real PII in git history (Leads_SBL_Tampa.xlsx)
2. Dialer logs every call as CONNECTED — call data is corrupted
3. Rate limiting fails open + IP bucket uses "unknown" key in many configs
4. No email-token invite flow; admin must set plaintext password in API body
5. Analytics page caps at 100 leads client-side — actively wrong data for larger orgs

3. ARCHITECTURE BREAKDOWN
================================================================================

Frontend Architecture
  src/app/<route>/page.tsx          — Next.js App Router pages ("use client")
  src/features/<feature>/components — Feature-scoped React components
  src/components/layout/            — DashboardLayout, Sidebar, Header
  src/components/ui/                — Primitive components (@base-ui/react wrapped)
  src/app/_trpc/                    — tRPC client + React Query provider

  DISCREPANCY: CLAUDE.md says "shadcn/ui" but src/components/ui/ wraps @base-ui/react
  primitives (MUI's Base UI), not Radix UI. Different library, different API guarantees.

Backend Architecture
  /api/trpc/[trpc]/route.ts    — Next.js Route Handler, tRPC HTTP server
  src/server/trpc.ts           — Context creation, procedure factories
  src/server/api/root.ts       — Root router (9 namespaces)
  src/features/*/server/       — Feature routers
  src/server/teams/scope.ts    — Cross-cutting lead scope resolver
  src/server/activity.ts       — Activity logging helper
  src/server/authz.ts          — Role check helpers
  src/server/scraper/          — Python subprocess orchestration

Auth Flow
  Registration → org created → ADMIN user created
  Sign-in → NextAuth CredentialsProvider → bcrypt compare → JWT issued
  Every request → proxy.ts (Next.js 16 middleware) → getToken() → allow/redirect
  tRPC context → getServerSession() → JWT fallback → DB revalidation (cached 60s)
  Deleted user → JWT revalidation returns null → {} token → UNAUTHORIZED

API Structure (tRPC namespaces)
  auth:      register, resetPassword, confirmResetPassword, updateProfile, deleteAccount
  leads:     getAll, getById, delete, bulkDelete, create, bulkCreate,
             updateTemperatureOverride, updateCallOutcome, toggleStar, assign,
             createNote, getNotes, deleteNote, getActivities, customOutcomes.*
  calls:     logCall, getForLead, getRecent
  tasks:     create, update, delete, getAll, getCalendar, getAllForLead,
             getDueToday, getOverdue
  dashboard: getKpiStats, sidebarCounts, getTeamStats, getMyStats
  teams:     list, organizationMembers, myTeam, activityFeed, memberDetail,
             create, update, delete, inviteUser, setMembership
  scraper:   config, list, getById, start, stop, delete, importResults, previewResults
  scoring:   getRules, upsertRule, deleteRule, resetToDefaults
  websites:  getForLead, generate, update, delete

  NOTE: CLAUDE.md procedure table is significantly out-of-date — many procedures
  exist that are not documented there.

State Management
  - Server state: TanStack Query via tRPC hooks
  - UI state: Local useState per component — no global store
  - Auth state: NextAuth useSession() / getServerSession()
  - localStorage: Sign-in page stores savedUsers[] for the account picker

Caching Strategy
  Cache                TTL    Key pattern
  ─────────────────────────────────────────────────────────────
  Auth snapshot        60s    auth:user:{userId}
  Dashboard KPI        60s    dashboard:kpi:{orgId}
  Dashboard sidebar    30s    dashboard:sidebar:{orgId}
  Team stats           60s    dashboard:team:{orgId}
  Lead scope           60s    scope:lead:{userId}

  Cache invalidation is correct and targeted:
  - invalidateAuthSnapshot(userId) on profile changes/deletes
  - invalidateLeadScope(userId) on team membership changes

Roles / Permissions
  UserRole: ADMIN > MANAGER > USER

  Role     Lead visibility              Team management    Notes
  ──────────────────────────────────────────────────────────────────────
  ADMIN    All org leads                Full CRUD
  MANAGER  Team leads (if team leader)  Read + member assign  Role ≠ leader
  USER     Own assigned leads           None

  NOTE: MANAGER role is defined but isManagerOrAdmin() is only used in task/note
  edit+delete. Lead visibility difference comes from Team.leaderId, not UserRole.

Deployment
  - Docker Compose: postgres:16 + migrate (prisma db push) + app (standalone Next.js)
  - next.config.ts: output: 'standalone' — correct for containerized deployment
  - No Redis in docker-compose.yml — must be added manually for rate limiting

Realtime / Queues
  NONE. No WebSockets, SSE, or job queues. Scraper status is polled from client.
  Scraper runner uses globalThis for hot-reload survival but dies on server restart.

4. DATABASE + PRISMA AUDIT
================================================================================

Schema Quality: 7/10

WELL DONE:
  - Cascade deletes correct: Lead → CallLog/Note/Activity/Task (cascade);
    assignedTo FK is SetNull (correct)
  - Password reset tokens store only SHA-256 hash (never raw token)
  - Composite indexes on all common query patterns
  - twilioCallSid unique constraint prepared for future Twilio integration

PROBLEMS:

  [HIGH] Dual-state Task model
    completed Boolean @default(false)
    status    TaskStatus @default(PENDING)  // PENDING | IN_PROGRESS | COMPLETED
    Two fields tracking the same thing. Every update needs sync logic.
    Recommendation: Remove completed boolean; derive from status.

  [HIGH] Task.organizationId is nullable
    organizationId String?
    Tasks can exist without an org. getAll compensates with a user join but
    if the creator's org changes, the task's effective org changes silently.

  [HIGH] LeadTag missing constraints
    organizationId String   // bare String, no FK to Organization
    No @@unique([name, organizationId]). Duplicate tags possible.
    No cascade when Organization is deleted. Zero UI usage — zombie model.

  [MED] Activity.type is String not enum
    type String
    ActivityType union exists in activity.ts but DB accepts any string.
    A typo in logActivity silently creates unrecognized activity types.

  [MED] Pipeline + PipelineStage are completely unused
    Both models exist, Lead has pipelineStageId FK, but zero router procedures
    read or write pipeline data and zero UI exists.

  [MED] ScraperJob JSON-string array fields
    locations  String @default("[]")
    categories String @default("[]")
    Should be Json type with @default([]); using String requires manual
    JSON.stringify/parse in 4+ places.

  [MED] Note and Activity have no organizationId
    Must join through Lead for org filtering. Cannot efficiently query all
    org activities without a join. Will cause table scan issues at scale.

  [LOW] CallLog.disposition is unvalidated free-form String
    Never written by any production code path.

  [LOW] No @@unique on Team name within org
    Duplicate team names in the same org are silently allowed.

Missing Constraints
  - LeadTag: missing @@unique([name, organizationId])
  - LeadTag: missing proper FK to Organization
  - GeneratedWebsite: no org-level unique constraint

Risky Patterns
  - No migration history (prisma db push) — adding required fields without
    defaults will crash production
  - ScraperJob.logs: DB column has no VARCHAR limit; only capped in runner code

8. SECURITY AUDIT
================================================================================

WHAT IS WELL-DONE
  1. Constant-time bcrypt dummy hash — prevents user enumeration via timing attacks
     DUMMY_HASH in src/lib/auth.ts
  2. Password reset token stored as SHA-256 hash — token never in DB; atomic
     delete-on-use prevents replay
  3. JWT revalidation on every request — deleted users lose access within 60s
  4. Rate limiting on all auth mutations:
     - register: 10/hr/IP
     - resetPassword: 3/hr/email + 10/hr/IP
     - confirmResetPassword: 20/hr/IP
     - sign-in: 10/min/email
  5. Organization isolation — every org-scoped read/write includes organizationId filter
  6. Defense-in-depth on notes/activities — lead: { organizationId } join filter
  7. Middleware defaults to protected — proxy.ts protects all routes by default
  8. Zod validation on all inputs — max lengths, email format, URL format, bounds

SECURITY ISSUES

  [CRITICAL] Real PII in git history
    Leads_SBL_Tampa.xlsx at repo root contains real customer contact data.
    Even if removed from HEAD, it remains in git history indefinitely.
    Fix: git filter-repo --path Leads_SBL_Tampa.xlsx --invert-paths + force push

  [HIGH] Rate limiting IP detection broken in common deployments
    src/lib/rateLimit.ts:66-78:
      return headers.get("x-real-ip") ?? "unknown";  // when TRUSTED_PROXY not set
    When TRUSTED_PROXY is not set (default), all requests without x-real-ip share
    the key "ratelimit:auth:register:ip:unknown". In many reverse proxy setups
    (Nginx without extra config, direct Docker access), x-real-ip is not set.
    Result: all clients share one register bucket, or one attacker blocks all
    registrations for an hour.
    Fix: Document TRUSTED_PROXY prominently; default to x-real-ip regardless of
    TRUSTED_PROXY setting; use x-forwarded-for first hop only with TRUSTED_PROXY=true.

  [HIGH] Rate limiting fails open during Redis outage
    return { ok: true, remaining: limit, resetAt: ... };  // always ok when Redis down
    During a Redis outage, brute force on passwords is possible.
    Acceptable availability tradeoff but must be documented and monitored.

  [HIGH] Admin-set-password invite flow
    teams.inviteUser takes { password: z.string().min(8) }. Admin types another
    person's password directly into the app. Stored in browser history, request
    logs, and any API gateways.
    Fix: Implement email-token invite flow.

  [MED] deleteAccount does not handle org ownership
    If the deleted user was the only ADMIN of their org, the org now has no admin.
    Other users in that org become orphaned.
    Fix: Require ownership transfer or org deletion before account deletion.

  [MED] xlsx package vulnerability
    package.json: "xlsx": "^0.18.5"
    CVE-2023-30533 (prototype pollution). Package frozen at 0.18.5 on npm.
    Fix: Replace with exceljs or a maintained fork.

  [LOW] No CSRF on tRPC mutations
    NextAuth handles CSRF for its own routes. tRPC uses POST with application/json.
    Acceptable for API-only routes but should be documented.

9. PERFORMANCE + SCALE AUDIT
================================================================================

WHAT WORKS WELL AT SCALE
  - Redis caching of KPIs at org level (60s TTL)
  - Cursor pagination on leads, tasks, activities — no offset scans
  - $queryRaw for daily call aggregation (DB-level date_trunc, not 7 queries)
  - Per-request scope memoization (prevents duplicate Redis/DB calls)
  - createMany + $transaction for bulk lead import (N leads → 2 DB round-trips)
  - Composite indexes covering all common query patterns

PERFORMANCE ISSUES

  [HIGH] Analytics page caps at 100 leads client-side
    src/app/analytics/page.tsx:37
    Should use a server-side leads.groupBy({ by: ['status'] }) aggregation.

  [MED] Dialer fetches all leads then filters client-side
    src/app/dialer/page.tsx:30
    Fetches 100 leads, then leads.filter(l => l.phone) in JavaScript.
    Fix: Add phone: { not: null } filter to the query.

  [MED] reconcileOrphanedJobs() on every scraper list fetch
    Called on every /scraper page load — a DB write on every read.
    Fix: Run once on startup; or check reconciliation timestamp with a TTL.

  [MED] Dashboard getTeamStats fetches 200 activities in memory
    Uses take: 200 then loops in JavaScript to find "last active" per user.
    Fix: Use groupBy + _max(createdAt) for DB-side aggregation.

  [LOW] N+1 in lead assignment
    leads.assign: 3 separate round-trips when 2 would suffice.

SCALE PROJECTIONS

  100 users (~10-15 orgs)
    ✅ Solid. Redis caching + auth snapshot cache handle this comfortably.

  1,000 users (~100 orgs of 10 users)
    ⚠️ Mostly fine. Watch:
      - reconcileOrphanedJobs() creates unnecessary DB writes on every poll
      - Without Redis, every dashboard load hits DB (5 parallel queries per org)
      - getTeamStats 200-activity in-memory aggregation feels heavy with 50+ members

  10,000 users (~500 orgs of 20 users)
    ❌ Multiple bottlenecks:
      - Activity/Note tables without orgId column → expensive join-based range scans
      - Dashboard cache at 60s TTL → 500 simultaneous cold cache misses possible
      - Single PostgreSQL instance with no connection pooling config
      - Rate limiting IP key collisions when many users share "unknown" bucket

  100,000 users (~5,000 orgs)
    ❌❌ Architecture needs significant changes:
      - Single PostgreSQL needs PgBouncer + read replicas
      - Redis needs clustering or sentinel
      - ScraperJob.logs as Text in main DB → should be object storage (S3/R2)
      - Activity table unbounded growth (no TTL, no archival)
      - Scraper Python subprocess model cannot scale horizontally
      - No CDN for static assets

WHAT BREAKS FIRST
  1. Redis single-node at ~5,000 concurrent users
  2. Activity/Note join queries at ~50,000 leads per org
  3. Dashboard KPI cache stampede on TTL expiry
  4. Scraper Python process on auto-scaling infrastructure
